# ADR 002: Асинхронная обработка возвратов через RPA

## Контекст
В процессе интеграции с Kaspi API v2 было выявлено архитектурное ограничение со стороны маркетплейса: API не поддерживает нативную передачу заявок на возврат и не отдает ивенты возврата напрямую в реальном времени. Полагаться исключительно на поллинг флага `returnedToWarehouse` рискованно, так как это ведет к рассинхронизации ERP и фактического склада.

## Решение
Разработан гибридный подход (API + RPA):
1.  **RPA-скрипт (Парсер):** Написан изолированный headless-скрипт, который запускается по cron строго в 11:00 (учитывая SLA платформы, когда прекращается падение заявок на текущий день).
2.  **Нормализация данных:** Скрипт парсит заявки, отсекает суффиксы (например, `-1`) из ID заказа и сопоставляет внутренний код маркетплейса с нашей БД.
3.  **UI-Очередь:** Данные формируют внутреннюю очередь возвратов. Физический возврат товара оприходуется сотрудником склада через UI в один клик с перемещением в зону обработки.

## Последствия
*   **Позитивные:** Обеспечена 100% консистентность склада. Бизнес не теряет деньги на потерянных возвратах.
*   **Негативные:** Зависимость от верстки личного кабинета маркетплейса, требующая мониторинга работоспособности скрипта.
