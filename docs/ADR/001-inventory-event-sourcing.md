# ADR 001: Использование Event-подобной модели для складских остатков

## Контекст
При объеме 800-1000 заказов в сутки классический CRUD-подход к таблице остатков (прямая перезапись поля `quantity`) приводит к гонкам данных (race conditions) и невозможности отследить историческую консистентность. Кроме того, физическое списание товара происходит не в момент заказа, а в момент отгрузки курьеру.

## Решение
1.  **Разделение логики бронирования и списания:** Введена модель, где таблица `inventory_balances` хранит данные в разрезе `location_id` и `condition_state_id`. При статусе заказа `packing` изменяется только `reserved_quantity`, физический остаток не трогается.
2.  **Ledger транзакций:** Физическое списание (`quantity -= N`) происходит строго при переводе заказа в статус `shipped`. В этот же момент создается неизменяемая запись в `inventory_transactions`.
3.  **Изоляция:** Для защиты от race conditions используется `SELECT FOR UPDATE` при резервировании и списании.

## Последствия
*   **Позитивные:** Полный аудит движения товаров. Нулевой риск уйти в отрицательные остатки при параллельной обработке Celery-воркерами.
*   **Негативные:** Усложнение логики запросов для вычисления реального доступного остатка (`quantity - reserved_quantity`).
